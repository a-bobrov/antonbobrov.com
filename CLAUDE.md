# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A personal academic website built with **Hugo** + the **Wowchemy / Hugo Academic** theme (`starter-hugo-academic`), driven from R via the **blogdown** package. Deployed to Netlify; the live site is `https://antonbobrov.com/`.

The Wowchemy/Hugo Academic stack used here is the legacy (pre-HugoBlox rename) version, and the Hugo version is pinned to **0.104.3** — see "Hugo version pin" below.

## Common commands (run from R / RStudio)

The expected workflow is interactive R, not a CLI build:

```r
library(blogdown)
serve_site()    # live-reload local preview (LiveReload). Most-used command.
build_site()    # full local build into public/
stop_server()   # stop the serve_site() process
```

Helpers used in history: `usethis::git_push()` is sometimes used to push after edits.

Netlify itself builds the site by invoking `hugo` directly (see `netlify.toml`); there is no R step on the deploy server. If you ever need to reproduce the Netlify build locally, run `hugo` with Hugo 0.104.3 — newer Hugo versions break this theme.

There are no tests, linters, or CI in this repo.

## Hugo version pin (important)

Hugo **0.104.3** is pinned in two places and they must agree:

- `.Rprofile` — `options(blogdown.hugo.version = "0.104.3")` (controls `blogdown::install_hugo()` / `serve_site()`).
- `netlify.toml` — `HUGO_VERSION = '0.104.3'` (controls the deploy build).

Upgrading Hugo is not a one-line change: the Wowchemy theme bundled here was last released against Hugo ~0.104 and breaks on modern Hugo. If you upgrade Hugo, you almost certainly also need to migrate the theme (Wowchemy was renamed to **HugoBlox** and the module paths under `themes/github.com/wowchemy/...` changed).

## Repo layout (the non-obvious parts)

Standard Hugo directories (`content/`, `layouts/`, `static/`, `assets/`, `data/`, `public/`, `resources/`) behave as usual. The Hugo-Academic–specific structure:

- **`config/_default/`** — site config split across `config.yaml`, `params.yaml`, `menus.yaml`, `languages.yaml`. Edit `params.yaml` for appearance/SEO/features (analytics, search, comments, etc.). The theme is selected in `config.yaml` (`theme: "starter-hugo-academic"`).
- **`content/home/*.md`** — the homepage is built from **widgets**, one file per section. Two front-matter fields control them:
  - `active: true|false` — whether the widget renders.
  - `weight: <int>` — vertical order (lower = higher on page).
  Currently active widgets: `about`, `projects`, `publications`. (Heads-up: `demo-hero.md` has `active: flase` — a typo, not an intentional `false`. Treat it as disabled.)
- **`content/publication/<slug>/index.md`** — each paper is a Hugo **page bundle**: the folder name is the URL slug, `index.md` carries metadata (title, authors, abstract, `publication_types`, `featured`, `links`, `url_pdf`), and any PDF (e.g. `District_Dissent.pdf`) sits next to `index.md`. The Publications widget on the homepage and `/publication/` listing both pull from this folder.
- **`content/post/`**, **`content/event/`** (talks), **`content/project/`**, **`content/slides/`** — same page-bundle pattern as publications.
- **`content/authors/admin/_index.md`** — the site owner's bio, education, interests, and social links. This is what the About widget renders; edit here, not in `content/home/about.md`.
- **`themes/starter-hugo-academic/`** — the theme skin (mostly static + a `theme.toml`). The real templates live in the Wowchemy modules below.
- **`themes/github.com/wowchemy/wowchemy-hugo-themes/modules/`** — vendored Hugo modules (`wowchemy`, `wowchemy-core`, `wowchemy-plugin-netlify`, `wowchemy-plugin-netlify-cms`, `wowchemy-plugin-reveal`, `wowchemy-seo`). These are referenced by `module.imports` in `config.yaml` and resolved locally — there is no live `hugo mod get`. Do not delete this tree.
- **`layouts/partials/blocks/`** — project-level overrides of Wowchemy widget templates. Hugo's lookup picks project `layouts/` before module `layouts/`, so mirroring a widget's path here (e.g. `layouts/partials/blocks/features.html`) lets you patch a single line without forking the theme. Used to widen the Skills row from 3 to 4 columns (`col-sm-4` → `col-sm-3`). This is the cleanest way to tweak any Wowchemy widget on this abandoned theme.
- **`R/build.R`, `R/build2.R`** — blogdown hooks that run before/after Hugo. Currently empty (comments only); blogdown still sources them.
- **`index.Rmd`** — blogdown sentinel file (`site: blogdown:::blogdown_site`). Do not edit; do not delete.
- **`public/`** — Hugo build output. Committed to the repo but regenerated by every build; don't hand-edit.
- **`.Rprofile`** — project-local R options: sources `~/.Rprofile`, sets `blogdown.knit.on_save = TRUE`, `blogdown.method = 'html'`, and pins the Hugo version.

## Adding a publication (the common task)

1. Create `content/publication/<Slug>/index.md` with the same YAML shape as the existing entries (`title`, `date`, `publishDate`, `authors`, `publication_types`, `abstract`, `featured`, `publication`, `links:` and/or `url_pdf:`). Copy from `content/publication/Service-Quality/index.md` as a template.
2. Drop the PDF (if any) into the same folder and reference it via `url_pdf:` or under `links:`.
3. With `serve_site()` running, the listing on the homepage and `/publication/` updates automatically.

`publication_types` is a string-encoded enum from the theme — `"2"` = journal article / working paper, used throughout this site.

## Things to be careful about

- The Wowchemy ecosystem is abandoned/renamed. Do not blindly run `hugo mod get -u` or upgrade theme modules — there is no upstream that matches what's vendored here.
- `content/home/*.md` `weight` values are reused across widgets (e.g., several widgets are at `weight: 50`); when reordering, scan the whole directory rather than picking a number in isolation.
- The repo URL placeholder in `params.yaml` (`features.repository.url`) still contains literal angle brackets (`<a-bobrov>`/`<antonbobrov.com>`); leave it or replace it intentionally — it affects the "Edit this page" link in the theme.
